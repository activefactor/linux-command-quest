# Linux Command Quest 開発ルール

## 1. 目的

本ドキュメントは、Linux Command Questを開発する上での共通ルールを定義する。

このゲームは、Linuxコマンドを安全に学ぶための教育用シミュレーターである。開発では「本物らしさ」と「安全性」を両立し、学習者が安心して失敗できる環境を作ることを最優先とする。

## 2. 開発の基本方針

- 実際のOSコマンドは実行しない
- すべての操作はブラウザ内の仮想環境で完結させる
- Linuxらしい見た目、挙動、エラーメッセージを重視する
- 初学者が理解できる日本語補足を適切に表示する
- 学習UIとLinux画面を明確に分離する
- 危険操作はゲーム内で再現し、実害は絶対に発生させない
- 仕様、設計、セキュリティポリシーに反する実装は行わない

## 3. ドキュメント優先順位

仕様判断で迷った場合は、以下の順に優先する。

1. セキュリティポリシー
2. 要件定義書
3. 基本設計書
4. 提案書
5. 実装時の判断

安全性に関わる場合は、演出や利便性よりもセキュリティポリシーを優先する。

## 4. 技術選定ルール

### 4.1 MVP

MVPでは、できるだけ軽量な構成とする。

- HTML
- CSS
- JavaScriptまたはTypeScript
- Canvas API
- ブラウザ内メモリによる状態管理

### 4.2 導入してよいもの

- UI補助のための小規模ライブラリ
- テスト用ライブラリ
- Lint、Formatter、型チェックツール
- 開発サーバー

### 4.3 導入に慎重になるもの

- 本物のシェルを呼び出す仕組み
- サーバー側で任意コマンドを実行する仕組み
- 過剰に重いフレームワーク
- 学習体験より実装都合を優先する抽象化

## 5. コーディングルール

### 5.1 命名

- 変数名、関数名、クラス名は役割が分かる名前にする
- コマンド実装は `executePwd`、`executeLs` のように動詞を含める
- ミッション判定は `checkMissionGoals` のように意図が分かる名前にする
- 危険操作判定は `detectDangerousCommand` のように安全性が読み取れる名前にする

### 5.2 構造

MVPでは以下の責務に分ける。

- 画面描画
- 入力処理
- コマンド解析
- コマンド実行
- 仮想ファイルシステム
- ミッション管理
- ヒント管理
- インシデント判定
- スコア管理

1つの関数に複数の責務を詰め込まない。

### 5.3 コメント

- 処理の意図がコードから読み取れる場合、コメントは不要
- 仮想ファイルシステム、パス解決、危険操作判定など、誤解が事故につながる箇所には短いコメントを書く
- コメントで仕様を上書きしない

### 5.4 エラー処理

- 未対応コマンドは明確にエラー表示する
- 存在しないパス、ファイル、ディレクトリはLinux風エラーを返す
- 学習者向けの日本語補足を必要に応じて表示する
- 例外でゲーム全体が停止しないようにする

## 6. UI開発ルール

### 6.1 Linux画面

- 黒背景を基本にする
- 等幅フォントを使用する
- ターミナル出力はLinux風に表示する
- コマンド結果は学習UIではなくLinux画面に表示する
- CLIの雰囲気を壊す過剰な装飾は避ける

### 6.2 学習UI

- 問題文、ヒント、進捗、解説はLinux画面とは別レイヤーに表示する
- 半透明パネルなどで「ゲーム側の補助」であることが分かる見た目にする
- 長文は読みやすく行間を確保する
- 初学者向けの文言は短く、具体的にする

### 6.3 トップページ

- 通常のWebランディングページにはしない
- CLI起動画面として表示する
- 自動タイピング演出を入れる
- ASCIIアートでタイトルを表示する
- `start`、`manual`、`credits` などのコマンド入力で進行できるようにする

### 6.4 vi風マニュアル

- `vi manual.txt` で開ける
- 編集機能は実装しない
- インサートモードは実装しない
- `:q!` または `:wq` で閉じられる
- 戻り方は画面内テキストに明記する

## 7. コマンド実装ルール

### 7.1 実装範囲

MVPで対応するコマンドは以下とする。

- `pwd`
- `ls`
- `cd`
- `cat`
- `cp`
- `chmod`
- `vi`
- `clear`
- `help`

### 7.2 実行方針

- 入力文字列を直接実行しない
- コマンド名と引数を解析し、許可された関数に振り分ける
- 未対応オプションは未対応として返す
- Linuxの完全再現より、学習に必要な再現性を優先する

### 7.3 危険操作

危険操作は実行せず、ゲーム内イベントとして扱う。

例:

- `chmod 777`
- `chmod -R 777 /`
- `rm -rf /`
- 設定ファイルの削除
- バックアップ前の設定変更

MVPでは `rm` 自体を未対応またはミッション外禁止として扱う。

## 8. データ管理ルール

### 8.1 仮想ファイルシステム

- 仮想ファイルシステムはアプリ内部のデータとして保持する
- 実際のローカルファイルにはアクセスしない
- パス解決は専用関数を通す
- `..` や絶対パスの扱いを必ずテストする

### 8.2 ミッションデータ

- ミッションはデータとして定義する
- 達成条件、ヒント、危険操作ルールをミッションごとに分ける
- コードに直接条件を散らばらせない

### 8.3 学習履歴

MVPでは永続保存しない。将来的に保存する場合は、セキュリティポリシーに従って保存対象を最小限にする。

## 9. テストルール

### 9.1 優先してテストする箇所

- コマンドパーサー
- パス解決
- 仮想ファイルシステム操作
- ミッション達成判定
- 危険操作判定
- `vi` 閲覧モードの終了処理

### 9.2 手動確認

MVPでは、以下を手動で確認する。

- トップページの自動タイピング演出
- ASCIIアートの崩れ
- コマンド入力の体験
- 学習UIとLinux画面の視覚的分離
- 危険操作時の警告表示
- マニュアルから `:q!` と `:wq` で戻れること

## 10. レビュー観点

レビューでは以下を確認する。

- 実OSに影響する処理が入っていないか
- 任意コマンド実行につながる処理がないか
- 要件定義書のMVP範囲から逸脱していないか
- コマンド挙動が学習目的に合っているか
- 初学者にとってエラーやヒントが分かりやすいか
- UIがLinux画面と学習UIを混同させていないか

## 11. 禁止事項

- ユーザー入力を `eval`、`Function`、シェル、OSコマンドに渡すこと
- 実ファイルシステムを操作すること
- 実ネットワークに接続する演習をMVPに含めること
- 危険コマンドを実際に実行すること
- セキュリティ上の警告を単なる演出として無視すること
- 学習者が戻り方を理解できない画面遷移を作ること
