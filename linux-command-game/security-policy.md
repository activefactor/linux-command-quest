# Linux Command Quest セキュリティポリシー

## 1. 目的

本ポリシーは、Linux Command Questを安全に開発、運用するためのセキュリティ方針を定義する。

本システムはLinuxコマンドの学習を目的としているが、実際のOS、ファイル、ネットワーク、シェルには影響を与えない設計とする。学習者が危険なコマンドを入力しても、現実の環境に被害が発生しないことを最重要要件とする。

## 2. セキュリティ基本方針

- 実OSコマンドを実行しない
- 実ファイルシステムを操作しない
- ユーザー入力を安全なコマンドパーサーでのみ処理する
- 許可リスト方式で対応コマンドを制御する
- 危険操作はシミュレーションとして扱う
- 学習データや個人情報は最小限にする
- 将来的にサーバー連携する場合も、任意コード実行を許可しない

## 3. 保護対象

### 3.1 MVPで保護するもの

- 利用者のブラウザ環境
- 利用者のローカルファイル
- 利用者の入力内容
- アプリケーション内部状態
- ミッションデータ
- 仮想ファイルシステムデータ

### 3.2 将来的に保護するもの

- ユーザーアカウント
- 学習履歴
- スコア
- 研修管理データ
- 管理者用ミッションデータ
- サーバーAPI

## 4. 脅威モデル

### 4.1 想定する脅威

- 入力されたコマンドが実OSで実行される
- `rm -rf /` などの危険入力が現実の環境に影響する
- `eval` や動的コード実行により任意JavaScriptが実行される
- 仮想パスの処理ミスにより外部データへアクセスする
- XSSにより任意スクリプトが実行される
- ミッションデータ改ざんにより危険な挙動が追加される
- 学習履歴に不要な個人情報が保存される

### 4.2 MVPで想定しない脅威

MVPではサーバー機能、ログイン機能、外部通信を持たないため、以下は対象外とする。

- 認証突破
- サーバーサイドの権限昇格
- APIキー漏洩
- データベース侵害
- マルチテナント間の情報漏洩

ただし、将来的に機能追加する場合は再評価する。

## 5. 入力処理ポリシー

### 5.1 コマンド入力

- 入力文字列はシェルに渡さない
- 入力文字列は `eval`、`Function` に渡さない
- コマンド名と引数に分解して処理する
- 対応コマンドは許可リストで管理する
- 未対応コマンドはエラーとして扱う
- パイプ、リダイレクト、サブシェルはMVPでは未対応とする

### 5.2 許可リスト

MVPで許可するコマンドは以下のみとする。

- `pwd`
- `ls`
- `cd`
- `cat`
- `cp`
- `chmod`
- `vi`
- `clear`
- `help`

これ以外のコマンドは、文字列として受け取っても実行しない。

### 5.3 禁止する実装

以下の実装は禁止する。

```js
eval(userInput);
new Function(userInput);
child_process.exec(userInput);
child_process.spawn(userInput);
```

ブラウザMVPではNode.jsの `child_process` は使わない。

## 6. 仮想ファイルシステムポリシー

### 6.1 基本方針

- ファイル、ディレクトリ、権限、所有者はアプリ内データとして表現する
- 実際のローカルファイルにはアクセスしない
- パスは必ず正規化する
- 仮想ルート `/` の外へ出られないようにする
- `..` を含むパスでも仮想ルート外へ移動させない

### 6.2 ファイル操作

- `cp` は仮想ファイルノードのコピーとして実装する
- `chmod` は仮想ファイルノードの `mode` を変更するだけにする
- `cat` は仮想ファイルノードの `content` を表示するだけにする
- `vi` は閲覧専用とし、編集は行わない

### 6.3 危険操作

以下は実行ではなく、警告またはインシデントイベントとして処理する。

- `chmod 777`
- `chmod -R 777 /`
- `rm -rf /`
- システム設定ファイルの削除
- バックアップ前の設定変更

## 7. XSS対策

### 7.1 表示方針

- ユーザー入力をHTMLとして描画しない
- ターミナル出力はテキストとして描画する
- DOMに表示する場合は `textContent` を使用する
- `innerHTML` の使用は原則禁止する
- ミッション本文にHTMLを埋め込まない

### 7.2 Canvas描画

Canvasに描画する文字列も、コマンドとして再評価しない。Canvasは表示専用とする。

## 8. データ保存ポリシー

### 8.1 MVP

MVPでは学習履歴を永続保存しない。

保存する場合でも、ブラウザメモリ内の状態に限定する。

### 8.2 将来拡張

LocalStorageやサーバー保存を追加する場合は、以下を守る。

- 個人情報を保存しない、または最小限にする
- コマンド履歴の保存可否を明確にする
- 保存期間を定義する
- ユーザーが履歴を削除できるようにする
- 管理者画面には認証を設ける

## 9. 外部通信ポリシー

### 9.1 MVP

MVPでは外部通信を行わない。

- 実サーバーへSSHしない
- 実APIを呼び出さない
- 実ネットワーク診断を行わない
- `curl` などのネットワーク系コマンドは未対応とする

### 9.2 将来対応

ネットワーク演習を実装する場合も、実通信ではなくモックレスポンスを基本とする。

## 10. 依存関係ポリシー

- 依存ライブラリは必要最小限にする
- メンテナンスされているライブラリを選ぶ
- 不要な依存を追加しない
- 脆弱性情報を確認する
- ビルド成果物に不要な秘密情報を含めない

## 11. セキュリティテスト

### 11.1 必須確認

- 入力コマンドが実OSに渡されないこと
- `eval`、`Function` が使われていないこと
- `innerHTML` でユーザー入力を表示していないこと
- `..` で仮想ルート外に出られないこと
- `chmod 777` が警告扱いになること
- `rm -rf /` が実行されないこと
- `vi` でファイル編集ができないこと

### 11.2 テスト入力例

```text
rm -rf /
chmod -R 777 /
cd ../../../../../../
cat ../../etc/passwd
<script>alert(1)</script>
$(whoami)
`whoami`
pwd; rm -rf /
```

上記はすべて安全に無害化、未対応化、またはゲーム内警告として処理されること。

## 12. インシデント対応方針

開発中にセキュリティ上の問題を発見した場合は、機能追加より修正を優先する。

対応手順:

1. 問題の再現条件を記録する
2. 影響範囲を確認する
3. 一時的に該当機能を無効化する
4. 修正する
5. 再発防止のテストを追加する
6. 必要に応じて本ポリシーを更新する

## 13. 禁止事項

- 本物のLinuxコマンド実行機能をMVPに入れること
- 学習者の入力をそのままコードとして実行すること
- 実ファイルを読み書きすること
- 実ネットワークへアクセスすること
- 危険操作を「教材だから」という理由で実行可能にすること
- セキュリティ警告を後回しにして開発を進めること
